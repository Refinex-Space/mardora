# Vue2 Playground Design

## Goal

Add a second playground site for Markora based on Vue 2.6, Vue CLI 4, and Webpack 4. The site must match the existing `/playground` user-facing capabilities while staying isolated from the current Next.js playground.

## Context

The repository is a Bun workspace with Turborepo. The existing playground lives in `playground/react-playground/app/playground` and is implemented with Next.js, React, `@uiw/react-codemirror`, and direct `@refinex/markora/src` imports. The new playground should live in a separate workspace app and validate Markora in a legacy Vue CLI 4 + Webpack 4 integration environment.

## Constraints

- Use `vue@2.6.x`, `@vue/cli-service@4.x`, Webpack 4, and TypeScript.
- Add the new app under `playground/vue2-playground`.
- Do not add a separate `package-lock.json` or `pnpm-lock.yaml`.
- Do not refactor or share logic with the existing React playground.
- Depend on `@refinex/markora: "workspace:*"` and consume the package build output, not `@refinex/markora/src`.
- Expose familiar Vue CLI commands from the app package, including `npm run dev`, `npm run build`, and `npm run lint`.
- Keep localStorage keys separate from the existing React playground.

## Architecture

Create `playground/vue2-playground` as an independent workspace app. It will contain its own Vue CLI project files, TypeScript configuration, Webpack compatibility configuration, static assets, and Vue components.

The app will copy the existing playground's default Markdown examples and configuration defaults into its own source tree. This keeps the Vue2 playground free to evolve independently and avoids touching `playground/react-playground/app/playground`.

The app will import Markora from package exports such as `@refinex/markora`, `@refinex/markora/preview`, and `@refinex/markora/plugins` after `packages/markora` has been built. Documentation will state that developers should build or watch `packages/markora` before starting the Vue2 playground.

## Components

The top-level `Playground.vue` owns the main state:

- `contents`
- `currentContent`
- `mode`
- `config`
- `nodes`
- `output`
- `outputTime`
- `saveStatus`
- panel open states

Child components communicate with props and events:

- `Header`: panel toggles, save status, mode switch.
- `Sidebar`: document list, selection, create, delete, rename.
- `EditorPane`: CodeMirror editor, preview view, and generated output view.
- `Devbar`: plugin toggles, editor settings, preview settings, AST toggle, output timing.
- `Footer`: word, line, and character counts.
- `CreateContentDialog`: simple new-document interaction.

No Vuex or other global state library is needed.

## Feature Parity

The Vue2 playground must match the existing `/playground` capabilities:

- `live`, `view`, `code`, and `output` modes.
- Rich Markora editing in `live` mode.
- Source Markdown editing in `code` mode.
- Static HTML/CSS preview generated by `preview()` and `generateCSS()`.
- Generated HTML and CSS output display.
- Default sample documents.
- Create, select, delete, and rename documents.
- Debounced localStorage persistence.
- Plugin toggles generated from `allPlugins`.
- Editor option toggles for base styles, keybindings, history, tab indent, active-line highlighting, and line wrapping.
- Preview option toggles for base CSS inclusion and sanitization.
- AST node display controlled by the Devbar.
- Save status, output timing, and content counts.

## CodeMirror Integration

Vue2 will use CodeMirror 6 directly through `EditorView` and `EditorState`; it will not use React wrappers.

Editor instances are created in `mounted` and destroyed in `beforeDestroy`. The editor is rebuilt when changes to mode, theme, Markora config, or active plugins require a new extension set. Content-only changes should use CodeMirror dispatch where practical to avoid unnecessary instance churn.

Read-only output panes may also use CodeMirror instances so generated HTML and CSS remain close to the current playground experience.

## Data Flow

On startup, the app reads Vue2-specific localStorage keys:

- `markora-vue2-playground-contents`
- `markora-vue2-playground-current`
- `markora-vue2-playground-version`

If no saved state exists, the app seeds its copied default documents. If the saved version is stale, it refreshes default documents while preserving user-created entries.

Markdown changes debounce both state updates and localStorage writes. Save status transitions through `idle`, `saving`, and `saved`.

Active plugins are derived from `allPlugins` and the plugin toggle map. Preview and editor outputs are recalculated when content, mode, theme, plugins, or relevant config changes.

## Webpack 4 Compatibility

`vue.config.js` will hold compatibility configuration for Vue CLI 4 and Webpack 4. It should handle workspace package resolution and transpilation needs for modern dependencies used by Markora and CodeMirror.

The app must avoid Next.js, React, and React-specific UI dependencies. UI controls should be implemented locally with Vue templates and scoped CSS.

Update the root README with a short section for the Vue2 playground:

- how to run the app with familiar commands such as `npm run dev`, `npm run build`, and `npm run lint`.

## Verification

Implementation is complete only when:

- `packages/markora` builds successfully.
- The Vue2 playground installs through the repository dependency workflow without adding an independent lockfile.
- `npm run build` or the equivalent app build command succeeds in `playground/vue2-playground`.
- `npm run lint` or the equivalent app lint command succeeds in `playground/vue2-playground`.
- A browser check confirms document management, editing, preview, output, plugin toggles, config toggles, AST display, localStorage persistence, responsive panel behavior, and basic error-free rendering.

## Rollback

Rollback is limited to removing `playground/vue2-playground`, removing the README section for the Vue2 playground, and removing any workspace or Turbo references added only for this app.
